Silent Hill 4 — .debug Section Reverse-Engineering

Target: SLUS_208.73 · Silent Hill 4 E3 2004 Trial (SILENT_HILL_4_VER_E3) · PS2 / MIPS R5900 · analyzed 2026-06-05

Verdict. The .debug section is DWARF version 1 debugging information emitted by Metrowerks CodeWarrior (MW MIPS C Compiler 2.4.1.01) for PlayStation 2, paired with a DWARF-1 .line number program. It is fully intact and parseable. From it we can recover, mechanically and with high fidelity: 504 original source files and their directory tree, 8,833 function signatures with named & typed parameters and locals, 5,061 structs / 516 unions / 721 enums (52,036 typed/offset fields), 4,041 typed global variables at fixed addresses, and 155,679 line→address records. 3 · How CodeWarrior encodes DWARF 1 here (the reversed format)

Everything below was confirmed by reading raw bytes (tools/dwarf1/probe*.py). Little-endian throughout.

3.1 · DIE stream (.debug)

The section is a flat stream of Debugging Information Entries. There is no abbreviation table (that is a DWARF-2 concept); every DIE is self-describing:

DIE        := u32 length      ; total bytes of THIS die, including the length word
              u16 tag         ; TAG_* (compile_unit, structure_type, global_subroutine, ...)
              attribute*      ; until 'length' is consumed
attribute  := u16 (attr | form)   ; low nibble = FORM_*, high 12 bits = AT_*
              value           ; size determined by FORM
; Children are stored INLINE after a DIE; you find the next *sibling* via AT_sibling.
; A "null entry" (length < 6, usually the 4-byte word) terminates a sibling chain.

Forms (low nibble of the attribute word) — this is what makes a v1 reader easy; you can skip any unknown attribute:

FormValueEncoding
FORM_ADDR0x14-byte target address
FORM_REF0x24-byte offset of another DIE within .debug
FORM_BLOCK2 / BLOCK40x3 / 0x4u16/u32 length prefix, then raw bytes
FORM_DATA2 / DATA4 / DATA80x5 / 0x6 / 0x72 / 4 / 8 byte constant
FORM_STRING0x8inline NUL-terminated string

3.2 · Compile unit

One TAG_compile_unit per function/section (≈ one per function — that is why CU count ≈ function count). It carries AT_name = the full source path, AT_producer, AT_language (0x2 = C), and AT_low_pc/AT_high_pc/AT_stmt_list (offset into .line). The very first CU decodes as:

off 0x000  len 0x62  TAG_compile_unit
  AT_sibling   = 0x0000cc
  AT_low_pc    = 0x00100000
  AT_high_pc   = 0x00100228
  AT_stmt_list = 0x00000000        -> .line offset for this unit's line program
  AT_language  = 0x00008000        -> vendor (this CU is crt0.s / assembly)
  AT_producer  = "Metrowerks MW GAS R5900 Assembler"
  AT_name      = "V:\sce281\ee\lib\crt0.s"

3.3 · Type references & the pointer/const trick

CodeWarrior does not emit pointer_type or typedef DIEs (both counts are 0). Instead a member/parameter/variable names its type via one of:

AttributeFormMeaning
AT_fund_typeDATA2a fundamental type code (FT_int=0x8, FT_float=0xe, FT_void=0x14, …)
AT_user_def_typeREFoffset of a struct/union/enum/array/func DIE
AT_mod_fund_typeBLOCK2[modifier bytes…][u16 fund_type]
AT_mod_u_d_typeBLOCK2[modifier bytes…][u32 DIE-ref]

Modifier bytes (applied left→right): 0x01 = pointer-to, 0x02 = reference-to, 0x03 = const, 0x04 = volatile. So sfCldObject.stay_node stores AT_mod_u_d_type = 01 6a0d0000 → "pointer to the DIE at 0x0d6a" (which resolves to struct sgQTNode).

3.4 · Member offsets, globals, and locals (location atoms)

AT_location is a tiny stack program of atoms; we only ever see three shapes:

struct member  : 04 <u32 offset> 07      ; OP_CONST offset, OP_ADD   -> field byte offset
global var     : 03 <u32 address>        ; OP_ADDR addr            -> absolute load address
register param : 01 <u32 regno>          ; OP_REG  reg             -> arg in a register
frame local    : 02 <u32 reg> … 04 <off> 07 ; OP_BASEREG + OP_CONST  -> stack slot

Worked example — struct sfCldObject members decode to offsets +0x00, +0x30, +0x34, +0x4c, +0x4d, +0x50, matching AT_byte_size = 0x60.

3.5 · Enumerations

TAG_enumeration_type holds AT_byte_size and AT_element_list (BLOCK4) = repeated { value, "name\0" }; the value width follows AT_byte_size. e.g. enum sgIKSolveResult { SG_IK_RESULT_NONE=0, SG_IK_RESULT_OK=1, … }.

3.6 · Line program (.line)

per unit:  u32 total_length            ; bytes incl. this header
           u32 base_pc                 ; address the offsets are relative to
           entry* {
              u32 line                 ; source line number
              u16 column               ; 0xffff = "whole line / none"
              u32 pc_offset            ; add to base_pc
           }

8,831 line programs, 155,679 entries, covering PCs 0x00100000 … 0x01ee56bc (the entire main image and the room overlays).

7 · Applying it to Ghidra — and the EE-Reloaded question

7.1 · Why the EE-Reloaded STABS/.mdebug analyzer can't be pointed at this

EE Reloaded's symbolic analyzer targets .mdebug — the MIPS/SGI ECOFF symbolic header (HDRR): fixed-size record arrays (file descriptors FDR, procedure descriptors PDR, local/external symbols SYMR/EXTR, auxiliary type words AUXU, dense numbers, line tables), sometimes with GCC STABS strings embedded. That is a completely different byte format and data model from DWARF-1's length/tag/attribute DIE stream. This binary has no .mdebug section at all, so the analyzer simply has nothing to chew on — it cannot be "recycled" at the parsing level.

EE-Reloaded mdebug analyzerWhat SH4 needs
Section.mdebug (HDRR).debug + .line (DWARF 1)
ModelECOFF records / STABS stringsDIE tree, form-typed attributes
Reusable parser?No — incompatible formats
Reusable scaffolding?Yes — the Ghidra-side application layer (below)

What is worth recycling from it is the integration scaffolding, which is format-agnostic and already battle-tested for EE programs:

  • How it registers an Analyzer that triggers on an EE/PS2 ELF and reads non-loaded overlay blocks.
  • How it builds Ghidra DataTypes (structs/unions/enums/arrays/function-defs) into managed categories and resolves cyclic refs.
  • How it attaches function signatures, stack/register variable storage, and source-line comments.
  • Its handling of EE relocations and the per-room overlay address space.
Also note: Ghidra's built-in DWARF analyzer supports DWARF 2–5 only; it does not implement a v1 reader and will ignore this .debug. So neither EE-Reloaded nor stock Ghidra handles it out of the box — a purpose-built DWARF-1 reader is required either way. The good news is DWARF-1 is small and self-describing; the parser already exists in tools/dwarf1/.

7.2 · Recommended implementation

Three viable shapes; Option A is recommended, written so its parser core can later be lifted into Option B.

OptionShapeProsCons
A · Parser → JSON → applier recommended Pure-Python DWARF-1 reader (90% done here) emits a normalized JSON model; a thin in-process Ghidra/Jython script (run via run_ghidra_script) applies types+sigs+globals+line comments in one batch. Also emit a .gdt type archive. Iterate the hard part in Python; deterministic; reviewable JSON; one fast apply pass (no thousands of MCP round-trips); .gdt reusable on the PC binary. Two pieces to wire; Jython apply script to write.
B · Native analyzer extension Java AbstractAnalyzer packaged as its own Ghidra extension, structurally modeled on EE-Reloaded's mdebug analyzer; could be contributed back as a sibling "DWARF1" analyzer. One-click in the GUI; redistributable; native overlay/reloc handling. Java; slower to iterate; must learn EE-Reloaded internals; overkill for a one-off.
C · Type-archive only Python emits a .gdt + a names/signatures script; import into both the PS2 and PC databases. Maximally portable types; smallest surface. Leaves function/var application to a second step.

7.3 · The apply transform (what the importer does)

  • Types, two passes. Pass 1: create empty named shells for every struct/union/enum/array/func DIE (keyed by DIE offset) to satisfy forward and cyclic references. Pass 2: fill members — offset from the location atom, type resolved through fund/user/mod-blocks, pointers via PointerDataType, fundamentals → Ghidra builtins.
  • Functions. rename → set prototype/signature; map parameters to register storage (OP_REG) and locals to stack storage (OP_BASEREG+offset); stamp the source file as a namespace or plate comment.
  • Globals. apply the recovered data type at the OP_ADDR address.
  • Line info. from .line, set file.c:NN comments (or a source map) per address; enables grouping the listing by original file.
  • Validation. reconcile names against the 22,721-entry .symtab; assert recovered AT_byte_size equals the summed/aligned field layout.

8 · Porting to the PC (GOG) version

The PC port is the same Konami codebase recompiled for x86/Windows, so the recovered knowledge transfers; the addresses do not. Plan:

  • Types → ship the recovered structs/enums as a .gdt and import directly into the PC database. Layout/field names are essentially identical (watch for pointer width 4→4 — same; and any #pragma pack differences — unlikely within the engine).
  • Names/signatures → port by function matching: generate signatures from the (now fully-named) PS2 reference and match against the PC binary (Ghidra Function ID / BSim, IDA FLIRT/Lumina-style, or BinDiff). The 504-file source map makes triage tractable — match subsystem by subsystem.
  • Rosetta value → even unmatched functions become readable because you know the参与 types and the source module they belong to; reversing the PC build stops being "squint at FUN_00xxxxxx" and becomes "fill in en_hyena.c".
